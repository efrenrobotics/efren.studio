There's a difference between code that works and code that feels *right*. Both get the job done, but one of them makes you want to keep building.

## The invisible details

The best software products are full of details that most people will never consciously notice:

- The subtle easing curve on a modal animation
- The way a loading state gracefully transitions to content
- The consistent spacing rhythm across every page

These things don't show up in feature lists or sprint reviews. But they compound. They're the difference between software that feels polished and software that feels rushed.

## Craft is a practice

I don't think craftsmanship is a talent — it's a habit. It's the decision to spend an extra fifteen minutes getting the spacing right. It's the refusal to ship something you know is "good enough" when you can see exactly how to make it better.

```typescript
// Good enough
const gap = 16;

// Better — the spacing follows a deliberate scale
const gap = tokens.spacing[4]; // 16px from a consistent system
```

This isn't perfectionism. Perfectionism is paralyzing. Craft is progressive — each small improvement moves the whole forward.

## Why it matters early

Some people say you shouldn't worry about craft early in your career. Just ship, they say. Learn by doing.

I partially agree. You absolutely need to ship. But I think you can ship *and* care about how it's built. The habits you form early are the ones that stick.
